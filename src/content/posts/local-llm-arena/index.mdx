This post is trying to scratch an itch I've had for a while.
Comparing small local LLMs on programming tasks without going purely by feel.
This means the test cases are deterministic and easy to programatically run and verify.

To keep the scope manageble I went with the classic [LeetCode "invert a binary tree"](https://leetcode.com/problems/invert-binary-tree/description/) problem which let even [famous programmers fail](https://x.com/mxcl/status/608682016205344768).

The whole thing runs on local hardware.
No Colab, no API keys, just a GPU and a few hundred lines of Python.
You should be able to follow along by copy-pasting each block below.

## Prerequisites

- An NVIDIA GPU with recent CUDA drivers. Mine is a 5070 Ti, 16 GB. Anything that can hold a 4B Q4 quant works.
- Python 3.11+ with `huggingface_hub` (`pip install huggingface_hub`).
- llama.cpp built with CUDA. From a fresh checkout:

  ```sh
  git clone https://github.com/ggml-org/llama.cpp
  cd llama.cpp
  cmake -B build -DGGML_CUDA=ON
  cmake --build build --config Release -j
  ```

  The binary you want is `build/bin/llama-server`.
  If you have Unsloth Studio installed it drops its own copy at `~/.unsloth/llama.cpp/build/bin/llama-server`, which is what I use below.

The models are [Unsloth's GGUFs](https://huggingface.co/unsloth) of Gemma 4 E4B Instruct and Qwen3.5 4B, both at the dynamic `UD-Q4_K_XL` quant.
`huggingface_hub` will download them to `~/.cache/huggingface/` on first run, so no need to fetch them by hand.

## Calling the model

`llama-server` is llama.cpp's HTTP server.
Point it at a GGUF and a port, and it speaks the OpenAI chat-completions protocol:

```sh
llama-server -m gemma-4-E4B-it-UD-Q4_K_XL.gguf --port 8123 -c 8192 -ngl 99 -np 1
```

That separates model loading from orchestration.
The Python side stays a thin HTTP client:

<CodeFile code={callModelSrc} lang="python" filename="call_model.py" href={callModelHref} />

`temperature=0`, `top_k=1`, `seed=42`, and `-np 1` on the server pin the sampler.
On the same hardware and llama.cpp build, the same prompt produces byte-identical output every run.
The function also returns llama-server's per-call `timings` block (tokens generated, tokens per second, total milliseconds), which we use for the performance table later.

## Generating samples

`llama-server` loads one model at a time, so the script cycles it: start pointed at the next GGUF, send the prompt, capture what comes back, shut it down, repeat for each entry in the model list.
A background thread runs alongside each call, polling `/proc/<pid>/{status,stat}` and `nvidia-smi` to capture per-process RAM and CPU and per-GPU VRAM and utilization.

Set `LLAMA_BIN` at the top of the file to your `llama-server` path before running.

<CodeFile code={generateSrc} lang="python" filename="generate.py" href={generateHref} />

Run it from the same directory as `call_model.py`:

```sh
python3 generate.py
```

Each entry writes `solution.py`, `raw-response.txt`, and `metrics.json` to its own folder under `samples/`.

## The judge

The judge loads each `solution.py`, calls `invert_tree` against a small set of trees, and prints a row of ✓ or ✗ per model.
A `-` means the file or function was missing entirely.
