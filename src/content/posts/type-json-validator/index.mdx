There is a small thing that has bugged me for years.
I fetch some JSON from an API, I write a TypeScript type for the shape I expect back, and then at runtime the two are free to disagree.
The type was erased during compilation, so nothing actually checks that the data matches what I declared.

The usual answer is to reach for a library like [Zod](https://zod.dev/) and write the shape a second time as a schema.
Now I have a type and a schema saying the same thing, and it is on me to keep them in sync by hand.
That always felt backwards to me.

But TypeScript already knows how to check whether a value fits a type.
That is most of what the compiler does all day.
So I wanted to see how far I could get by borrowing that and writing no validator of my own.
I built this little playground (well, Claude did most of the typing) to find out.

It is live.
Pick a sample, or paste your own JSON into the value pane.
The type on top starts as a guess inferred from the data, and you edit it to tighten the shape.
The value below stays plain JSON, and it is rechecked against the type as you type.
It feels like JSON Schema, except the schema is a TypeScript type and the thing doing the checking is the real compiler.

## The compiler is the validator

There is no validator here, and the value pane is not really checked as JSON at all.
Behind it sits a one line TypeScript file that you never see:

```ts
const data: Root = /* your JSON, verbatim */;
```

JSON is valid TypeScript syntax, so your data drops straight into that initializer as a fresh object literal assigned to a typed target.
TypeScript then checks it structurally.
A missing field, a wrong primitive, or a `null` where the type does not allow one all turn into errors at the exact spot in the data.
And because it is a fresh literal, TypeScript also flags properties that are not in the type at all, which is the strict mode of a schema library for free.

The editors are [Monaco](https://microsoft.github.io/monaco-editor/), the same one VS Code is built on, running the real TypeScript language service in a web worker.
I ask that worker for the errors on the hidden file, shift each one up by the single header line, and paint it onto the JSON pane.
So the squiggles you see on plain JSON come from the actual compiler and not from anything I reimplemented.

## Where the starting type comes from

When you load JSON, I walk the parsed value and emit a first guess at `type Root`.
Arrays of objects are the fun case.
A key that shows up in some elements but not others becomes optional, and a field that is sometimes `null` becomes a `T | null`.

It is only a guess from one sample though.
If a field happened to be `null` in this particular response, the guess is `null` and nothing more.
You tighten it to `string | null` by hand, and the squiggles tell you right away whether the rest of the data still fits.

## Getting a real function back out

Squiggles are lovely while editing, but sometimes I want an actual function I can call.
That is the "Generate validator" button, and it is the part I find most interesting, so here is the whole thing.
The reader does not have to take the output on faith either.
Once a validator is generated, "Run it against the value above" runs it in your browser and tells you what it returned.

The idea is to ask the [TypeScript Compiler API](https://github.com/microsoft/TypeScript/wiki/Using-the-Compiler-API) for the resolved shape of `Root` and walk it.
There are four small steps: stand up a compiler, resolve the type, walk it into a string of checks, then strip the types so the string runs.

### A compiler with no files

The compiler normally reads from disk.
In the browser there is no disk, so I use [@typescript/vfs](https://www.npmjs.com/package/@typescript/vfs) to give it an in-memory filesystem with two files in it.

```ts
import ts from "typescript";
import { createSystem, createVirtualTypeScriptEnvironment } from "@typescript/vfs";

// A tiny standard library. The inferred types only use objects, arrays,
// primitives, and unions, so the checker just needs to know these globals
// exist. That keeps us off the network and out of the multi-megabyte real
// lib.d.ts files.
const MINIMAL_LIB = `
interface Array<T> { length: number; [index: number]: T; }
interface Boolean {}
interface Number {}
interface String { readonly length: number; }
interface Object {}
interface Function {}
interface IArguments {}
interface RegExp {}
`;

function makeChecker(typeText: string) {
  const files = new Map<string, string>();
  files.set("/lib.d.ts", MINIMAL_LIB);
  files.set("/index.ts", typeText);

  const system = createSystem(files);
  const env = createVirtualTypeScriptEnvironment(
    system,
    ["/lib.d.ts", "/index.ts"],
    ts,
    { target: ts.ScriptTarget.ES2020, strict: true, noLib: true },
  );

  const program = env.languageService.getProgram()!;
  return {
    checker: program.getTypeChecker(),
    source: program.getSourceFile("/index.ts")!,
  };
}
```

The `noLib` option turns off the real standard library, and the minimal stub takes its place.
The checker only needs to know that `Array` and a handful of globals exist for `string[]` to mean something.

### From a name to a resolved type

Now I find the `Root` declaration in the source and ask the checker what it actually is.

```ts
function findRoot(checker: ts.TypeChecker, source: ts.SourceFile) {
  for (const statement of source.statements) {
    if (ts.isTypeAliasDeclaration(statement) && statement.name.text === "Root") {
      const symbol = checker.getSymbolAtLocation(statement.name)!;
      return checker.getDeclaredTypeOfSymbol(symbol);
    }
  }
}
```

This is the step that pays for the whole compiler.
`getDeclaredTypeOfSymbol` hands back the resolved type, with `string[]` already understood as an array of strings and `A | B` already flattened into a union.
I never have to parse type syntax myself.
I just ask what it resolved to.

### Walking the type into checks

With a resolved type in hand, I walk it and build a string of boolean checks.
A `Type` carries a bitmask of flags for the primitives, and the checker has helpers for the rest.

```ts
function check(type: ts.Type, expr: string, checker: ts.TypeChecker): string {
  const F = ts.TypeFlags;

  if (type.flags & F.String) return `typeof ${expr} === "string"`;
  if (type.flags & F.Number) return `typeof ${expr} === "number"`;
  if (type.flags & F.Boolean) return `typeof ${expr} === "boolean"`;
  if (type.flags & F.Null) return `${expr} === null`;

  if (type.isUnion()) {
    const parts = type.types.map((t) => check(t, expr, checker));
    return `(${parts.join(" || ")})`;
  }

  if (checker.isArrayType(type)) {
    const element = checker.getTypeArguments(type as ts.TypeReference)[0];
    return `Array.isArray(${expr}) && ${expr}.every((item) => ${check(element, "item", checker)})`;
  }

  if (type.flags & F.Object) {
    const checks = checker.getPropertiesOfType(type).map((prop) => {
      const propType = checker.getTypeOfSymbolAtLocation(prop, prop.valueDeclaration!);
      const access = `${expr}.${prop.name}`;
      const optional = prop.flags & ts.SymbolFlags.Optional;
      const inner = check(propType, access, checker);
      return optional ? `(${access} === undefined || ${inner})` : inner;
    });
    return [`typeof ${expr} === "object"`, `${expr} !== null`, ...checks].join(" && ");
  }

  return "true";
}
```

Every branch is the checker doing the work.
`isUnion` gives me the members, `isArrayType` plus `getTypeArguments` gives me the element type, and `getPropertiesOfType` plus `getTypeOfSymbolAtLocation` give me each field and its type.
Optional fields show up as a flag on the property symbol, so they are allowed to be missing.
The real version in the demo handles a few more cases, like string and number literals and dropping `undefined` from unions, but this is the shape of it.

Wrapping that walk in a function body gives the validator you see in the output:

```ts
function validate(value: unknown): value is Root {
  const v = value as any;
  return /* the walked checks */;
}
```

### Stripping the types so it runs

That output is still TypeScript.
It has `: unknown` and a `value is Root` predicate, which the browser cannot run.
Rather than strip the annotations with a fragile regex, I hand the string back to the compiler and let it erase the types for me.

```ts
const js = ts.transpile(code, { target: ts.ScriptTarget.ES2020 });
const validate = new Function(`${js}\nreturn validate;`)();
validate(myJson); // true or false
```

That is exactly what the "Run it against the value above" button does.

Put together it is the whole generator.
Stand up a checker, resolve `Root`, walk it into a string, and transpile the string so it runs.
About 150 lines, and most of them are the walk.
This is the part a schema library cannot do from a type alone.
The type is read, not copied.

## Where it breaks

This is a toy, not a library.
The type guesser handles the JSON shapes I actually run into and not tuples, recursive types, generics, or index signatures.
The generated validator covers objects, arrays, unions, primitives, optionals, and `null`, and stops there.

Replacing Zod was never the point.
I just wanted to see that a TypeScript type already carries enough to validate real data, and that the compiler will hand that back to you if you ask nicely.
